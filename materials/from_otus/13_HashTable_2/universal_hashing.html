Функции хеширования¶

Хеширование делением¶

  • Числовое значение ключа делится на размер хеш-таблицы: hash(k)=kmodMhash(k)=kmodM
  • Хорошо работает в ситуации, когда ключи взяты из равномерного распределения
  • Такое способ не очень быстр (потому что деление - "медленная" операция)

Проблемы выбора M

  • MM не должно быть степенью двойки. Иначе для M=2pM=2p хеш будет просто pp младших битов ключа (что допустимо только в случае, если несколько последних знаков числа распраделены равномерно)
  • Хороши простые числа, не очень близкие к степени 2.
  • Идущие подряд значения ключей "порождают" идущие подряд значения хешей. Это может быть как плюсом, так и минусом алгоритма.

Сравните с 10й системой счисления при делении на 10k10k: нет неожиданности, что в остатке будет kk наименее значащих цифр изначального числа. То же верно и для систем с основанием 2.

Хеширование при помощи умножения¶

  • hash(k)=⌊(AkmodW) / (W/M)⌋hash(k)=⌊(AkmodW) / (W/M)⌋
  • AA взаимно простое с WW

  • на практике удобно W=2w,M=2mW=2w,M=2m, ww - размер машинного слова (16, 32, 64, как правило) Тогда хеш-функция превращается в hash(k)=⌊(Akmod2w) / 2w−m⌋hash(k)=⌊(Akmod2w) / 2w−m⌋, которую можно записать как простую функцию со "сдвигом":

    1  hash(key, A, M):
2      return A * x >> (w - M)

  • A⋅kmod2wA⋅kmod2w - аналогично хешированию делением, но AA вносит "возмущение" в ключ kk

  • Операция "сдвига" раскладывается на две операции:
    • собственно "деление" на 2(w−m)2(w−m)
    • удаление дробной части происходит естественным образом за счет "сдвига"

Такой подход очень быстр из-за простоты базовых операций (выполняющихся быстрее деления).

Более того, подобная функция близка к универсальной, если A - случайное нечетное число из {1,...,2w−1}{1,...,2w−1}. Про это мы еще поговорим.

Вопрос: почему нечетное?

Хеш-функции для строк и последовательностей¶

  • хеш должен зависеть от каждого символа
  • и хеш должен зависеть от каждого символа по-разному!
  • строки с одинаковыми символами в разном порядке, желательно, должны хешироваться по-разному

Пример простой хеш-функции для последовательностей: хеш-функция Пирсона.

1  PearsonHash(S, Table):
2     h := 0
3     for (i = 0; i < len(S); i++):
4         h := Table[h xor S[i]]  // Выбор значения из таблицы Table
5     return h

- Что такое Table? Это перемешанный список чисел 0..255. Он может быть как случайно перемешан, так и подобран специальным образом, чтобы обеспечить идеальное хеширование для определенного набора данных (см. далее)

Свойства хеш-функции Пирсона:

  • простой код и очень быстрая работа
  • две строки, различающиеся на один символ, никогда не создадут коллизии
  • Модифицировав работу с Table, моджно применять для архитектур с большим, чем 8, машинным словом
1  PearsonHash64(S, Table):
2     for i in 0..7:
3         h = Table[(S[0] + i) % 256]  // +i для учета сдвига
4         for each item in S:
5             h = Table[h xor item]
6         hh[i] = h
7     return concatenated hh

ARX (add-rotate-xor)¶

Быстрые и используемые на практике наборы функций для хеширования последовательностей.

Базируются на следующем принципе:

  • Сложение двух чисел по моудлю
  • Сдвиг элементов
  • XOR

Варианты ARX-алгоритма SipHash используются во многих языках, в том числе, python3.4+, Ruby, Perl.

Схемы адресации¶

Квадратичный пробинг¶

  • hash(k,i)=(hash′(k)+с1i+с2i2)modMhash(k,i)=(hash′(k)+с1i+с2i2)modM.
  • Лучше линейного, но нужно подбирать с1,с2,Mс1,с2,M.

Несколько популярных вариантов выбора констант

  • hash(k)=(hash′(k)+i2)modMhash(k)=(hash′(k)+i2)modM, где c1=0,c2=1,M−простое>3c1=0,c2=1,M−простое>3, фактор заполнения α<1/2α<1/2
  • hash(k)=(hash′(k)+(i+i2) / 2)modMhash(k)=(hash′(k)+(i+i2) / 2)modM, где c1=c2=1/2,M=2kc1=c2=1/2,M=2k
  • hash(k)=(hash′(k)+−1i⋅i2)modMhash(k)=(hash′(k)+−1i⋅i2)modM, где M≡3 mod 4M≡3 mod 4

Почему α<1/2α<1/2? Пусть есть сдивги xx и yy, указывающие на одну локацию, но x≠yx≠y, и 0≤x,y≤M/20≤x,y≤M/2.

hash(k)+x2≡hash(k)+y2modMhash(k)+x2≡hash(k)+y2modM
x2≡y2modMx2≡y2modM
x2−y2≡0 modMx2−y2≡0 modM
(x−y)⋅(x+y)≡0 modM(x−y)⋅(x+y)≡0 modM

x−y≠0,x+y≠0x−y≠0,x+y≠0 - значит, существует M/2 различных мест для записи.

Чем квадратичный пробинг лучше линейного?

Равномерность распределения хешей по таблице зависит от количества возможных вариантов обхода при пробинге.

  • Максимально M!M! вариантов обхода, столько требуется для "честного" равномерного хеширования
  • Для линейного пробинга - всего MM вариантов, по числу стартовых точек
  • Для квадратичного пробинга так же всего MM, но сами последовательности пробинга распределены более равномерно

Двойное хеширование¶

Идея: использовать 2 хеш-функции, чтобы получить M2M2 последовательностей пробинга.

Формально, это

hash(k,i)=(hash1(k)+i⋅hash2(k))modMhash(k,i)=(hash1(k)+i⋅hash2(k))modM

Для обхода всех ячеек при пробинге hash2(k)hash2(k) должна всегда возвращать взаимно простое с MM число. Простой способ сделать это:

  • Выбирать размер хеш-таблицы как степень 2: M=2pM=2p
  • hash2hash2 всегда возвращает нечетное число
  • К сожалению, на практике, если размер таблицы ≠2p≠2p, выбрать функцию может быть тяжело

Тем не менее, C# HashTable использует двойное хеширование

Количество попыток при пробинге¶

  • В случае, если распределение ключей близко к равномерному (хорошая хеш-функция и пробинг)
  • фактор заполненности таблицы равен α<1α<1,

Количество попыток пробинга равно в среднем: 11−α11−α

  • на практике все может быть хуже :)

Универсальное хеширование¶

Универсальное хеширование случайно выбирает хеш-функцию из семейтсва хеш-функций, чтобы осложнить жизнь злоумышленнику (или просто избежать ситуации с "плохими" данными).

H - семейство хеш-функций, отображающих ключи во множество {0..M−1}{0..M−1}. При этом, для пары ключей k,lk,l количество хеш-функций, для которых hash(k)=hash(l)hash(k)=hash(l), не более ||/M|H|/M

  • то есть шанс коллизии для случайно выбранной хеш-функции и двух случайных ключей не более 1/M1/M
  • это позволяет получить близкое к идеальному поведение

Построение класса универсальных хеш-функций

  • Выберем pp достаточно большое, чтобы любой ключ kk попадал в диапазон от 0 до p−1p−1
  • ℤp={0,1..p−1}Zp={0,1..p−1}, ℤ∗p={1,2..p−1}Zp∗={1,2..p−1} - множества чисел
  • hashab(k)=((ak+b)modp)modMhashab(k)=((ak+b)modp)modM - хеш-функция
  • ={hashab:a∈ℤp∧b∈ℤ∗p}H={hashab:a∈Zp∧b∈Zp∗} - всего p⋅(p−1)p⋅(p−1) функций

Как получилось, что это универсальные хеш-функции

r=(ak+b)modpr=(ak+b)modp
s=(al+b)modps=(al+b)modp

r−s≡a(k−l) (modp)r−s≡a(k−l) (modp) - так как pp - простое, и (k−l),a(k−l),a не равны 0 по модулю pp.

Поэтому коллизий (ak+b)modp(ak+b)modp не возникает. Более того, все пары (r,s)(r,s) будут различными для каждой пары (a,b)(a,b).

Осталось проверить, что коллизий не возникает и при взятииmodMmodM. Поскольку разные r,sr,s равновероятны, вероятность коллизии kиlkиl равна вероятности того, что r≡s(mod M)r≡s(mod M). Ее можно вычислить так:

⌈p/m⌉−1≤((p+m−1)/m)−1=(p−1)/m⌈p/m⌉−1≤((p+m−1)/m)−1=(p−1)/m

- количество значений для rr таких, что s≠rs≠r и s≡r(mod M)s≡r(mod M). Поскольку есть (p−1)(p−1) возможных s, вероятность будет

((p−1)/m)(p−1)=1/m,((p−1)/m)(p−1)=1/m,
что удовлетворяет требованию к универсальной хеш-функции.
